# System Architecture

## Overview

A publish-subscribe notification service built on raw TCP sockets with SSL/TLS encryption. A central broker manages client connections, topic subscriptions, and message routing.

---

## Socket Lifecycle

### Broker (Server)
```
socket() → setsockopt() → bind() → listen()
  └── ssl.wrap_socket()
      └── accept() [loop]
          └── thread: recv() → dispatch() → sendall()
```

### Client (Publisher / Subscriber)
```
socket() → connect()
  └── ssl.wrap_socket()
      └── sendall() / recv()
```

---

## Concurrency Model

- **One thread per client** on the broker using `threading.Thread`
- A `threading.Lock` protects the shared subscription map for all reads and writes
- Subscribers run a **dedicated listener thread** for receiving async `NOTIFY` frames, separate from the input REPL thread — both share the same SSL socket (reads and writes are on separate halves of the connection)

---

## SSL/TLS

- Self-signed certificate generated by `certs/gen_certs.py` using the `cryptography` library
- Server loads cert + private key; clients verify using the same cert as a CA
- Minimum TLS version enforced: **TLS 1.2**
- Cipher negotiated at runtime (typically `ECDHE-RSA-AES256-GCM-SHA384`)

```

    │                      │                          │
    │──PUBLISH sports msg──►│                          │
    │                      │  lock subscription map   │
    │                      │  find all subs for topic │
    │                      │──NOTIFY sports msg───────►│
    │◄──OK published to N──│                          │
```
